# React 스타일 가이드라인

## 컴포넌트 정의

- 컴포넌트는 **`function` 키워드 + PascalCase**로 작성. 화살표 함수 컴포넌트 금지 (TS 가이드의 "화살표 함수 기본" 원칙의 예외).
  - 이유: PascalCase는 JSX 문법 요구사항(소문자 시작 시 React가 HTML 태그로 취급), 제네릭 컴포넌트 문법 자연스러움, 호이스팅으로 "공개 API → 내부 구현" Top-down 구조 가능, 컴포넌트 경계와 내부 로직의 시각적 구분.
- 컴포넌트 내부 로직(이벤트 핸들러, 콜백, 헬퍼)은 **화살표 함수**로 작성.

## 핸들러 네이밍

- 컴포넌트 내부 핸들러: `handleXxx`
- props로 전달되는 이벤트 콜백: `onXxx`
- 내부 핸들러와 외부 prop이 같은 이벤트를 다루면 뒷부분 이름을 맞춘다 (`handleSubmit` ↔ `onSubmit`).
- `onXxx`는 **이벤트/사용자 액션 반응에만** 사용. 렌더 함수(`renderItem`), 데이터 함수(`fetcher`), 팩토리 함수에는 `on` 접두사 금지.

## 상태 관리

- **State Colocation**: 상태는 실제 사용하는 컴포넌트에 가장 가깝게 둔다. 형제 간 **동일한 값**을 함께 보거나 동기화해야 할 때만 가장 가까운 공통 부모로 lifting. "혹시 나중에 필요할까봐" 미리 올리지 않는다.
- **상태 그룹핑**: 항상 같이 변하는 값은 하나의 `useState`로 묶고(폼, 좌표 등), 독립적으로 변하는 값은 분리.
- 복잡한 상태 전이는 `useReducer` 고려.

## useEffect

- Effect는 **외부 시스템과의 동기화에만** 사용 (네트워크 요청/구독, 타이머, 이벤트 리스너, 브라우저 API, 서드파티 인스턴스).
  - 판별 시그널: **cleanup이 필요하면 거의 항상 Effect가 맞다. cleanup이 없는 Effect는 다시 검토.**
- Effect 작성 전 다음 순서로 검토:
  1. 사용자 액션 결과인가? → 이벤트 핸들러로 (부모 콜백 호출도 여기서)
  2. 다른 state/props에서 파생되는 값인가? → 렌더 중 직접 계산
  3. props 변경 시 state 초기화? → `key` prop 또는 렌더 중 계산
  4. 위 모두 아닌 외부 시스템 동기화인가? → 그때 Effect 사용
- **하나의 Effect는 하나의 책임만**. 의존성 배열이 책임별로 다르면 분리.

## 커스텀 훅

- 하나의 훅은 하나의 관심사만. 훅 이름을 한 문장으로 설명할 수 있어야 한다 ("~와 ~를 다룬다"가 되면 쪼갠다).
- 만능 훅(`useUserDashboard` 등) 금지. 작은 훅을 조합해서 사용.

## 메모이제이션 (useMemo / useCallback)

- 기본은 **사용하지 않는다**. 측정된 병목에만 적용.
- 적용 조건 (하나 이상 충족):
  - React DevTools Profiler로 병목 측정됨
  - 동일 계산이 빈번히 반복되며 실제 비용이 큼 (대규모 정렬·필터링 등)
  - `React.memo`로 감싼 하위 컴포넌트의 리렌더를 명확히 줄여줌
- 적용 금지:
  - 단순 연산(`.length`, 산술, 문자열 연결)
  - 자식이 `React.memo`가 아닌데 `useCallback`만 적용
  - "혹시 몰라서" 감싸기
- **React Compiler가 적용된 프로젝트면 수동 메모이제이션 사용 금지.** 예외: 외부 라이브러리가 참조 동일성(`===`)으로 변경 감지하는 경우(TanStack Query의 `queryKey` 등), Compiler가 bail-out한 위치(린트 경고가 알려줌 — 먼저 React 규칙 위반을 고치는 게 우선).

## 상태 공유 / Props Drilling

값의 **영향 범위**로 도구를 선택한다:

- 부모 → 직접 자식: 그냥 props
- 중간 컴포넌트가 값을 쓰지 않고 전달만 함: **Composition (children / Slot)** 우선
- 특정 서브트리 내부에서만 공유: **Context**
- 앱 전반의 독립된 지점들에서 구독: **전역 상태 관리 도구** (Zustand 등)

### Composition (children / Slot) 우선

- 중간 컴포넌트가 props를 쓰지 않고 자식에게 전달만 할 때는 Context보다 먼저 `children` / Slot 패턴으로 풀 수 있는지 검토.
- 중간 컴포넌트가 자식 prop 타입을 알 필요가 없어져 결합도가 낮아지고, 자식에 새 prop이 추가돼도 중간 컴포넌트는 수정 불필요.

### Context 사용 시

- Compound Component 내부 통신(Tabs, Accordion, Select 등) — 사실상 유일한 선택지
- 서브트리 국소 공유 상태
- 안정적·변경 드문 앱 수준 값 (테마, locale, 현재 사용자)
- **반드시 커스텀 훅으로 감싸서** null 체크와 에러 메시지를 캡슐화
- 자주 바뀌는 값을 넓은 Provider로 감싸지 말 것 (Provider 하위 모든 소비자 리렌더)

### 전역 상태 관리 도구 사용 시

- Selector로 필요한 부분만 구독 (전체 스토어 구독 금지)
